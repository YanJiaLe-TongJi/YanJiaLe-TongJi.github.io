项目复盘

物理实验报告生成器开发随笔与经验回顾

解决痛点，不等于一定能做成好产品。

2026 年 4 月 1 日 Word 原稿整理发布

这篇文章记录了我开发“物理实验报告生成器”项目的动机、技术方案、上线后的真实数据，以及一次并不轻松但很有价值的复盘。它既是项目经验总结，也是我第一次正式写下来的博客文章。

为什么会做这个项目

大一第一学期，物理实验报告成了我感受非常强烈的一类重复劳动。大量时间被消耗在复制粘贴学习通上的资料、在 Word 里艰难输入公式，以及重复整理那些变化并不大的结构化内容上。那段时间又正好是各种 Agent 工作流被广泛讨论的时期，于是我萌生了一个很直接的想法：能不能自己搭一个 Agent 工作流，把物理实验报告自动化完成掉？

一开始它只是一个给自己用的个人工具。但做着做着，我觉得也许能顺手“造福”一下身边的同学，于是这个项目逐渐从个人自动化工具转向了一个面向他人的 Web 应用。

项目时间线

  • 2026.2.9：打完美赛后正式开工，把模板和资料拉到本地，并把 Word 模板迁移到 TeX。
  • 2026.2.10：确定模型使用 Kimi K2.5，因为当时有免费的 15 元额度。
  • 2026.2.13：搭建好前端 HTML。
  • 2026.2.21：中间因为出去过年，开发暂停一周后继续。
  • 2026.2.25：初步完成前后端，测试后发现生成功能基本可用。
  • 2026.2.26：决定做成 Web 应用，开始开发登录管理等框架。
  • 2026.3.1：此前一直没用 Git 做版本管理，被 AI 改坏代码后艰难修复，从这天开始认真使用 Git。
  • 2026.3.3：租了云服务器和域名，项目正式上线，站点为 `reportgenerator.cn`。

技术实现其实并不是最难的部分

单从技术层面看，这个项目并没有太多特别复杂的难点。为了方便开发，后端用的是 Python Flask，前端是 JavaScript，外围管理系统也是常见的 Web 应用套路。在现在这个阶段，有 AI 的辅助，很多常规框架代码确实可以很快搭起来。

在 Agent 部分，我选择了 Kimi K2.5，主要原因也很现实：注册账号后送了 15 元的 API 额度，足够让我先把主要流程跑起来。

真正比较有意思的点有两个。第一，为了缓解长上下文下模型幻觉严重的问题，我没有让整篇报告完全串行生成，而是采用“串行 + 并行”的结构。很多章节，比如实验原理、实验方法和仪器等，本质上主要依赖实验资料，而不是依赖整篇上下文，因此这些部分可以拆开并行生成。之后再生成核心部分，比如基于用户上传数据的数据处理，再并行去完成思考题、误差分析和结论。

第二个是工程化优化。我发现用户每次生成报告时，都会重复解析同样的实验资料，这个过程既浪费时间，也增加调用成本。于是我把全部实验资料预先上传到后台，并调用视觉模型做统一解析，让用户在实际使用时能直接复用这部分结果，从而明显缩短生成时间。

上线后的真实结果

截至 2026 年 4 月 1 日，网站一共有 83 个注册用户。客观来说，对于这样一个项目，这个数字不算太差。但真正让我在意的是另一个数据：这 83 个注册用户里，只有大约三分之一配置了 API Key。也就是说，真正使用过这个项目的人，最多也只有三分之一。

这个转化率比我预期低很多。因为这个工具瞄准的是一个非常明确、非常真实的痛点：本校大一理工科学生每周都要完成繁琐的实验报告，而报告中除了数据处理部分之外，大量内容都带有明显的复制粘贴性质。学校提供的资料还是图片形式，很多时候还要先识图提取文字，再重新整理到文档里，整个过程相当耗费时间。

按照我最开始的想法，这种工具上线后应该很容易获得几百位用户，而且使用粘性应该比较高。但事实上，这个判断并没有成立。上线一个月后，我对这个落差做了几轮复盘。

为什么“痛点明确”却没有带来理想增长

第一，项目本身还不够完善

受限于 AI 输出的不稳定性、Word 公式渲染、实际上课内容与资料之间的偏差，生成出来的报告并不总能直接交付。很多时候仍然需要人工修改。也就是说，它并没有真正完成自动化，而只是完成了一大部分自动化。

第二，这个领域本身具有敏感性

按学校明面上的要求，实验报告应该由学生独立完成。虽然几乎所有人都默认会使用 AI 辅助，但当 AI 变成“全面接管报告”，而且这件事还直接和成绩挂钩时，大多数人还是会天然保持谨慎。这个心理门槛不是靠“功能存在”就能消除的。

第三，易用度仍然不够

我后来越来越觉得，“只有三分之一用户配置 API Key”是一个很关键的信号。虽然整个配置过程其实不复杂，说明也写得比较完整，不到两分钟就能完成，但对很多抱着“先看看”的用户来说，这已经足够构成劝退。很多人并不是在认真比较你的方案优劣，而是只要发现它不能开箱即用，就会直接离开。

理论上，一个解决方案是让所有用户直接走我的 API Key，但这个项目本身面向学生，又涉及相对敏感的使用场景，我很难向同学收取 token 成本。这意味着这个看似简单的问题，其实并不容易被彻底解决。

第四，真正容易传播的是“产品结果”，不是“生产工具”

物理实验报告有一个非常关键的属性：它高度重复、机械、可复制。这正是我觉得写它很痛苦的原因，也是我做这个项目的出发点。但反过来看，重复就意味着复用性极强。只要有一个人已经写好了一篇报告，他完全可以把成品分享给室友或同学，他们只要简单改一改数据，就能很快完成自己的版本。

换句话说，我的工具解决的是从 0 到 0.8 的工作，最后那 0.2 仍然要人工完成；但当第一个“1”已经被产出之后，继续传播和改写的成本，也许只有 0.1。于是工具本身只服务于少数人，而由工具生产出来的结果，反而可以更广泛地传播。

如果把这次经验浓缩成一句话，那就是：解决了痛点，不代表就做成了一个会自然增长的好产品。

最后的几点体会

  1. AI 编程确实是趋势，但会用和能用好是两回事。 这个项目主要用 Cursor 开发，大量代码由 AI 生成。AI 很强大，但真正要把它用好，还是需要更好的工作流和控制习惯。我的经验是，多用 plan mode，尽量提前写文档，少做大范围结构重构，同时善用 Git 做版本管理。当然，选择一个足够强的模型仍然是前提。
  2. 大部分用户不会主动给你反馈。 我最开始发公告时，希望大家遇到问题可以来反馈，甚至直接在 GitHub 提交代码。但现实是，大多数人发现不好用，会直接选择离开，而不是等你迭代。所以，上线前测试的重要性，比我原先理解得更高。

这是我的第一篇博客，整体更偏随笔和感想。写下这些内容，不只是为了记录一个项目，也是在提醒自己：技术能解决很多问题，但产品是否成立，还要面对更复杂的真实世界。
