项目复盘

物理实验报告生成器开发随笔与经验回顾

——解决痛点不等于好产品

2026 年 4 月 1 日 Word 原稿整理发布

为什么做这个项目：

大一第一学期，高频重复地完成物理实验报告这件事让我感到非常痛苦，也明显意识到它在消耗大量时间。那段时间里，我大约有百分之八十的精力都花在复制粘贴学习通上的实验资料、在 Word 里艰难输入公式，以及处理那些高度重复的文档劳动上。恰好当时又是各种 Agent 在互联网上被密集讨论和炒作的阶段，于是我萌生了一个很直接的想法：自己搭一套 Agent 工作流，能不能把物理实验报告这件事自动化掉？

一开始，这只是一个服务于我自己的个人工具。但做着做着，我又产生了“既然都做了，不如顺手造福一下同学”的念头，于是项目慢慢从个人自动化工具转向了一个面向他人的 Web 应用。

项目时间线：

  • 2026.2.9 : 打完美赛，正式开工，把各类模版和资料拉到本地，并把word模版迁移到tex
  • 2026.2.10: 确定模型使用kimi k2.5，因为有免费的15块额度
  • 2026.2.13: 搭建好前端html
  • 2026.2.21: 前一周去外面过年玩了，开发暂停了一周，接着开始。
  • 2026.2.25: 初步完成前后端，测试生成功能基本可用
  • 2026.2.26: 决定做成web应用，开发登陆管理等框架
  • 2026.3.1 : 一直没用git做版本管理，AI把代码改坏了，艰难修复，开始使用git做版本管理
  • 2026.3.3：租了个云服务器和域名，正式上线（reportgenerator.cn）。主要流程完成，剩下都是小修小补。

项目技术概览：

单从技术实现上看，这个项目并没有特别复杂的难点。为了方便开发，后端使用的是 Python 的 Flask 框架，前端则是 JavaScript；外围的管理系统也基本都是比较常规的 Web 应用模块。在现在这个阶段，很多这类基础代码在 AI 的帮助下都能较快搭起来，所以真正拉开差距的，并不是“会不会写页面和接口”，而是工作流和产品形态本身。

在 Agent 模型的选择上，我使用的是 Kimi K2.5，原因也非常现实：注册后会送 15 块钱的 API Key 额度，足够我先把主要流程搭起来并跑通。

其中一个我自己觉得还算有意思的小优化，是为了缓解长上下文下模型幻觉比较严重的问题，生成流程并没有采用完全串行，而是使用了“串行 + 并行”的方式。因为报告中的很多章节，比如实验原理、实验方法、实验仪器等，主要依赖实验资料本身，并不依赖前后文，所以这些部分可以拆开分别喂给模型，让每个请求只负责一块内容。等并行章节完成之后，再去生成核心部分，也就是根据用户上传数据得到的数据处理部分；随后再并行生成思考题、误差分析和结论，从而兼顾质量和速度。

另外还有一个偏工程化的优化。因为我发现，用户每次生成报告时，都会重复解析同样的实验资料，这是一种明显的重复成本。于是我把所有实验资料统一上传到后台，并调用视觉模型提前完成解析，让这部分工作在用户真正开始生成之前就已经完成，从而大幅缩短了整体生成时间。

更具体的技术细节，后续我会整理到个人 GitHub 主页里。

项目实际落地成果：

截至目前（2026 年 4 月 1 日），网站共有 83 个注册用户。客观来讲，对于这样一个项目，这个成绩不算差。但我注意到，这 83 名注册用户里，只有大约三分之一配置了 API Key。换句话说，在所有注册用户里，真正使用过这个项目的人，最多也只有三分之一。

而这个转化率在我看来其实是偏低的，因为这个工具瞄准的，确实是本校大一理工科同学一个非常真实的痛点：每周都要完成繁琐的实验报告，而除了数据处理之外，很多内容本质上都是重复、机械的复制粘贴；再加上学校提供的大量资料是图片形式，往往还要先识图提取文字，再重新整理进文档，整个过程相当耗费时间。

按我最开始的预估，这样的工具上线之后应该能比较轻松地获得数百位用户，而且用户粘性也理应不错。但事实并非如此。项目上线一个月后，围绕这个落差，我做了以下几轮复盘：

第一：该项目本身并不足够完善：

受限于 AI 本身的不稳定性、Word 公式渲染的问题，以及实际上课内容和资料之间可能存在的偏差，生成出来的报告并不总能达到“直接可交付”的要求，很多时候仍然需要人工修改。也就是说，这个项目并没有真正完成自动化，而只是完成了自动化的一大部分。

第二：项目本身领域敏感

按照学校明面上的要求，所有实验报告都需要学生自己完成。虽然绝大部分同学都在使用 AI 做某种程度的辅助，但当 AI 从“辅助”变成“全面接管报告”，而且这件事还直接和课程成绩相关时，大多数人还是会天然地保持谨慎和怀疑。

第三：易用度不够

我后来越来越觉得，“配置 API Key 的用户只占注册用户的三分之一”其实是一个很关键的信号。虽然整个配置流程并不复杂，我也写了说明，实际操作不到两分钟就能完成，但很多人本来就是带着“先试试看”的好奇心来的，一旦发现它不能开箱即用，热情就会立刻下降。这种相对偏高的使用门槛无疑劝退了很多人，毕竟大部分人天然怕麻烦，尝试新事物的意愿并没有我们想象中那么强，哪怕对象是大学生也是如此。

理论上，一个简单的解决方案是让所有用户都直接使用我的 API Key。但问题在于，这个项目面向的是学生群体，而且使用场景本身比较敏感，我很难把 token 成本转嫁给同学。所以对这个项目来说，这恰恰是一个看得见、但并不容易真正解决的问题。

第四：容易传播的是产品而不是工具本身

物理实验报告还有一个非常关键的特性：它高度重复、机械。这个特性一方面是我觉得写报告很痛苦的原因，也是我开发这个项目的出发点；但另一方面，重复也意味着它极其容易复制和复用。只要有一个人已经写好了一份报告，他就完全可以分享给室友或同学，对方只需要简单改一改数据，就能完成自己的版本，工作量甚至比使用我的工具还低。

换句话说，我的工具解决的是从 0 到 0.8 的工作，剩下的 0.2 仍然需要人工完成；但一旦已经有了第一个“1”，后续继续传播、分发、改写的成本可能只剩下 0.1。于是，工具本身只服务于少数人，而用工具生产出来的产品结果，却可以更广泛地传播。

所以如果要用一句话概括这次经验，那就是：解决痛点，并不等于就能自然长成一个好产品。

最后再谈谈一些别的体悟：

  1. AI 编程毫无疑问是趋势。本项目使用 Cursor 开发，大量代码都由 AI 生成。AI 很强，但“会用 AI”和“把 AI 用好”之间还是有很大差别。我的经验是，多用 plan mode，尽量把项目文档写清楚，尽可能少做大范围结构修改，同时一定要善用 Git 做版本管理。当然，选择一个足够好的模型，比如 GPT 或 Opus，本身就是一切的基础。
  2. 另外一个很现实的体会是：大部分用户其实不会主动给你反馈。项目刚开始上线时，我发过公告，希望大家遇到问题能反馈给我，甚至欢迎直接在 GitHub 提交代码。但事实是，大多数人如果发现不好用，会直接选择离开，而不是留下来等你迭代。所以，上线前测试的重要性，比我原来想象得更高。
